# Сайт клуба дзюдо «Локомотив» — план разработки

## Контекст

Клуб дзюдо «Локомотив» хочет заменить устаревший сайт (по анализу из майндмапа: нет подстраниц, слабая навигация, нет мобильной адаптации) на современный сайт с полноценной CMS-админкой. Ключевые требования:

- **Единая CMS+админка** — не два отдельных приложения, а одно место, откуда админы постят новости и редактируют сайт.
- **Интеграции с Telegram и ВК**: автоподтяжка постов из клубного TG-канала и группы ВК на сайт, без дублирования, если один и тот же пост опубликован в оба канала; обратный постинг с сайта минимум в Telegram (желательно и в ВК).
- **Супер современный дизайн**: много графики/анимации, но сайт должен оставаться быстрым, полностью адаптивным.
- **Стек**: полностью бесплатный/open-source, максимум кастомизации, без конструкторов и шаблонов — уникальный дизайн с нуля.
- **VPS Ubuntu 24.04** уже арендован, доступ будет передан позже.
- Магазин (кимоно/мерч) — сознательно вынесен за скобки v1.
- TG-канал и группа ВК у клуба уже есть, но доступ к созданию бота/API-приложения пока не готов — интеграции спроектированы как **опциональный, подключаемый позже слой**, не блокирующий остальную разработку.

**Важно про модели**: план составлен на Sonnet. Пользователь явно попросил, чтобы непосредственная **реализация (написание кода) велась на модели Opus** — при переходе к исполнению плана нужно переключить модель сессии на Opus (или делегировать фазы через Agent/Task с `model: opus`).

Проект гринфилдовый — репозиторий создан пустым специально для этого сайта, не связан с другими проектами пользователя.

## Технологический стек

| Слой | Технология | Почему |
|---|---|---|
| Сайт + рендеринг | **Next.js 15 (App Router)**, TypeScript | React-экосистема даёт максимум свободы для кастомного богатого UI/анимаций без ограничений шаблонизатора; `next/image`, стриминг, ISR — из коробки быстрый рендеринг при насыщенной графике |
| CMS/админка | **Payload CMS 3.x**, смонтированная в тот же Next.js-процесс на `/admin` | С версии 3 Payload — не отдельный сервер, а часть Next.js-приложения: один процесс, один билд, один Docker-контейнер. Это единственный из рассмотренных вариантов (сравнивали со Strapi+отдельный фронтенд и Django+Wagtail), который одновременно даёт «CMS = админка в одном приложении», MIT-лицензию, self-host и не сковывает дизайн публичной части шаблонами |
| БД | **PostgreSQL 16** через встроенный Postgres-адаптер Payload | Реляционные данные (тренеры↔группы, источники постов), надёжность, бесплатно |
| Стили | **Tailwind CSS 4** + кастомные компоненты | Полная свобода дизайна, никакого UI-конструктора |
| Анимации | **Framer Motion (Motion for React)** | Точечные анимации на GPU-композитных свойствах, не блокируют main thread |
| Медиа | Локальный диск VPS + `sharp` (ресайз/WebP/AVIF) | Не требует платного объектного хранилища на этом масштабе |
| Реверс-прокси/TLS | **Caddy 2** (авто Let's Encrypt) | Тот же паттерн, что уже использовался у пользователя в проекте ChickenFlow — знакомо и проверенно |
| Контейнеризация | **Docker Compose** (app + postgres + caddy, без Redis) | Redis не нужен на этом масштабе — Payload Jobs Queue работает поверх Postgres |
| CI | GitHub Actions (free tier): lint + typecheck + build | Бесплатно, деплой на прод остаётся ручным shell-скриптом |

Все компоненты — MIT/Apache/PostgreSQL-лицензия, self-hosted, ноль платных SaaS в цепочке.

## Модель данных (Payload-коллекции)

- **`Pages`** — блочный конструктор (`layout: blocks[]`) для Главной и «О клубе»: `HeroSlider`, `MissionBlock`, `CallToAction`, `PartnersStrip`, `Timeline` (История), `RulesList` (Заповеди), `TeamGrid` (Тренеры/Спортсмены), `VideoEmbed`, `Gallery`, `ScheduleTable`, `ContactForm`, `FAQAccordion`, `Statistics`, `RichText`. Даёт редакторам пересобирать страницы без разработчика.
- **`Coaches`** (Тренеры), **`Athletes`** (Спортсмены, включая служебное поле `parentalConsentObtained` для 152-ФЗ)
- **`ScheduleEntry`** (Расписание): группа, день/время, тренер, зал, возраст/уровень
- **`NewsPost`** (Новости) — центральная коллекция для интеграций: `sources[]` ({platform: site/telegram/vk, externalId, url, rawPayload}), `originPlatform`, `crosspostStatus` (трекинг обратной публикации), `needsReviewDuplicate`, `mergedFrom[]`
- **`MediaGallery`** (Фотогалерея/Фильмы/Интервью) — видео только embed (YouTube/RuTube/VK Video), не хостим файлы сами
- **`Partners`**, **`Media`** (upload-коллекция с авто-thumbnail через sharp)
- **`Users`** с ролями `admin` / `editor` / `coach` (access control в коде коллекций: coach правит только свой профиль и расписание своих групп)
- **`SocialPostQueue`** (служебная, не в основной навигации) — буфер сырых постов из TG/VK для пайплайна дедупликации
- **Globals**: `SiteSettings` (контакты, соцссылки, генеральный партнёр, SEO по умолчанию) и `IntegrationSettings` (токены TG/VK, флаги `enabled`/`crosspostOnPublish`, пороги дедупликации) — единая точка включения интеграций

## Интеграции Telegram / ВК

Архитектурный принцип: весь интеграционный слой — **опциональная надстройка**. Пока `IntegrationSettings.telegram.enabled` / `.vk.enabled` = `false` (по умолчанию), вебхуки отвечают 200 и ничего не делают, hooks обратного постинга не срабатывают. Сайт и CMS полностью работоспособны без единого токена — это учитывает то, что доступ к TG/VK API у клуба сейчас не готов.

- **Telegram**: Bot API webhook на `channel_post` (`/api/webhooks/telegram`). **Ограничение, которое нужно явно объяснить пользователю**: Bot API не даёт доступа к истории канала — бот видит только посты, опубликованные после того, как его сделали админом. Разовый импорт истории (если понадобится) — отдельная ручная задача через MTProto-сессию (GramJS), не часть постоянной инфраструктуры.
- **ВК**: **VK Callback API (webhook на `wall_post_new`)**, а не поллинг — обоснование: HTTPS-сервер уже поднят под Telegram-вебхук, событийная модель даёт мгновенную публикацию анонсов и не расходует квоты API так, как регулярный поллинг.
- **Обратный постинг** (сайт → TG обязательно, → ВК желательно): Payload hook `afterChange` на `NewsPost` при публикации (только если `originPlatform === 'site'`, чтобы не зациклить). Асинхронно через Payload Jobs Queue с ретраями, статус виден редактору в карточке поста.
- **Дедупликация кросс-постов** (пайплайн в `SocialPostQueue`): временное окно ±30 мин → нормализация текста (без разметки/эмодзи/ссылок) → text similarity (Jaccard/Dice по словам, порог 0.82) → усиливающий сигнал — pHash сравнение картинок → решающее правило: выше порога — авто-мерж в один `NewsPost` с обоими источниками; пограничная зона (0.5–0.82) — флаг `needsReviewDuplicate` и экран «Возможные дубликаты» в админке с ручным merge/reject; ниже 0.5 — публикуются как разные новости. Идемпотентность по `externalId` защищает от повторной доставки одного и того же вебхук-события.

## Деплой (VPS Ubuntu 24.04)

Docker Compose из трёх сервисов: `app` (Next.js/Payload), `postgres`, `caddy` (авто-TLS, реверс-прокси). Redis не нужен на этом масштабе. Ежедневные бэкапы `pg_dump` + tar медиа-директории с ротацией (14 дней + помесячно), с обязательной проверкой реального восстановления из бэкапа хотя бы раз перед релизом. Деплой — простой `deploy.sh` (`git pull && docker compose build && docker compose up -d` + health-check).

## Производительность и дизайн

`next/image` + `sharp` (WebP/AVIF, responsive srcset, lazy loading), видео-слайдер на poster-кадре с подгрузкой через `IntersectionObserver`, ISR для страниц новостей/контента с ревалидацией по Payload-хуку при публикации, самохостинг шрифтов через `next/font` (кириллица+латиница), Framer Motion только на transform/opacity (GPU-композит), уважение `prefers-reduced-motion`. Контрольная точка на каждой фазе с публичным UI — Lighthouse Performance ≥ 90 mobile.

Цветовая палитра/бренд-айдентика клуба не заданы явно в майндмапе («цвета дзюдо» — общая идея) — в фазе публичного дизайна (Фаза 2) нужно либо получить от пользователя логотип/брендбук клуба, либо предложить палитру на утверждение перед вёрсткой.

## Комплаенс РФ (152-ФЗ)

Страница «Политика конфиденциальности» в футере, обязательный (не преднастановленный) чекбокс согласия на обработку персданных на всех формах со ссылкой на политику, серверная валидация. Служебные поля `parentalConsentObtained`/`parentalConsentDate` в `Athletes` как административная подстраховка перед публикацией фото несовершеннолетних (само согласие собирается офлайн клубом).

## Критические файлы/директории

- `collections/NewsPost.ts` — `sources[]`, `crosspostStatus`, hooks обратного постинга и триггера дедупликации
- `globals/IntegrationSettings.ts` — единая точка включения TG/ВК, токены, пороги дедупликации
- `src/integrations/telegram/webhook.ts` — `/api/webhooks/telegram`
- `src/integrations/vk/webhook.ts` — `/api/webhooks/vk` (включая `confirmation`-ответ)
- `src/integrations/deduplication/pipeline.ts` — нормализация/similarity/pHash/решающее правило
- `blocks/*` — React-компоненты всех типов блоков `Pages.layout`
- `docker-compose.yml` + `Caddyfile` — топология прод-деплоя

## Верификация по ходу реализации

- Фаза 0: `npm run dev`, логин в `/admin`, `docker compose build`
- Фаза 1: ручное создание контента каждой коллекции под разными ролями, проверка access control
- Фаза 2: визуальная проверка на 3 брейкпоинтах в браузере (dev-сервер + preview), Lighthouse mobile ≥90, отправка тестовой заявки через контактную форму
- Фаза 3/4: тестовый TG-бот/ВК-сообщество разработчика — публикация поста и проверка появления на сайте за ожидаемое время; публикация новости в CMS и проверка появления в канале/группе
- Фаза 5: одновременная публикация одинакового текста в тестовые TG+ВК — на сайте должна остаться одна новость с двумя источниками; юнит-тесты на пороговые случаи similarity
- Фаза 6: полный прогон деплоя на staging/прод VPS, тестовое восстановление БД из бэкапа, прод Lighthouse, проверка что токены не отдаются в публичном API не-admin роли
